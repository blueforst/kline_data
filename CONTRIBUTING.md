# 贡献指南

## 项目结构管理

### 文件归档规范

当需要重构或替换现有代码时，请遵循以下步骤：

1. **创建改进版本**：
   - 在同一目录下创建新文件
   - 实现并测试新功能

2. **归档旧文件**：
   ```bash
   # 将旧文件移动到归档文件夹并添加日期后缀
   mv path/to/old_file.py archived/old_file_archived_$(date +%Y%m%d).py
   ```

3. **重命名新文件**：
   ```bash
   # 如果新文件名称不同，重命名为原文件名
   mv path/to/new_file.py path/to/old_file.py
   ```

4. **更新导入引用**：
   - 检查所有导入该模块的文件
   - 更新import语句
   - 更新文档中的引用

5. **运行测试**：
   ```bash
   pytest tests/
   ```

### 代码重构检查清单

- [ ] 新代码功能完整且经过测试
- [ ] 所有导入引用已更新
- [ ] 文档已更新
- [ ] 测试通过
- [ ] 旧文件已归档到 `archived/` 目录
- [ ] 在归档 README 中记录归档原因和日期

## 开发规范

### 代码风格

- 遵循 PEP 8 规范
- 使用类型注解
- 添加文档字符串（Google style）

### 提交规范

- 使用清晰的提交信息
- 一次提交只做一件事
- 提交前运行测试

### 分层架构原则

1. **配置层**：只负责配置管理，不包含业务逻辑
2. **存储层**：只负责数据存储和检索，不做计算
3. **读取层**：只负责数据读取，不做转换
4. **重采样层**：只负责时间周期转换
5. **指标层**：只负责技术指标计算
6. **SDK层**：提供统一的编程接口
7. **服务层**：提供REST API接口
8. **CLI层**：提供命令行工具

每层只依赖下层，不跨层调用。
